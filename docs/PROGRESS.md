## TL;DR

**Current stage: Stage 2 complete + hardened (code), proven on real data.** Stage 0 (marketing site), Stage 1 (three-variant template), and all of Stage 2 are built: the mockup generator (manual-brief + **places mode**: Google Places + Instagram-light → SalonBrief), the Supabase **`mockups` table**, and the **`mock.revivo.nl` SSR app** that renders a salon at a shareable `/{slug}` URL through the customer-template variants. The full loop is validated against **real** infra + a **real** prospect: a live Google Place (Utrecht Hairstyle) → Claude (OpenRouter) → grounded SiteConfig → Supabase → rendered in the mock app. A 5-lens adversarial audit of that first real mockup drove a **Stage 2.5 hardening pass** (the generator no longer fabricates KvK/BTW/email/booking-URLs; real lat/lng/rating ride through as structured grounding), and the mock app is **deployed** (Vercel, `revivo-mockups.vercel.app/{slug}`). Most recently, finding the mockup factually wrong about the salon's own business drove **Stage 2.6 — real-data sourcing**: a deterministic Treatwell scraper (`window.__state__` → typed `ListingFacts`, no LLM) plus `applyListingFacts`, which overwrites the config's factual fields (services/prices/hours/team/reputation/real reviews/real photos/booking) with the scraped truth while the LLM authors only voice. Verified end-to-end on Utrecht Hairstyle (33/33 ground-truth scraper checks; real LLM run grounded correctly). **Phase 5 + 5.5 done:** all three variants render `team` + `reputation` + real reviews + real photos in their own DNA, and the **3-template acceptance test passes** (atelier/studio/neon all correct on the Utrecht real-data config — 0 picsum, 0 fabrication). **Phase 6 done — and the measurement killed the planned design:** the screenshot-vs-screenshot vision comparator was built and *measured* on a mockup whose every fact was hand-verified correct against Treatwell — it produced **3/3 false positives** (misread price €55→€35, the Sunday "gesloten" cell, and the real phone), so it's **shelved as a manual spot-check, not a runtime gate**; the deterministic scrape measured accurate. The real residual risk (invented *prose*, the "Spaanse muziek" class) is now caught by a cheap, **validated** text check (`checkAboutFidelity`) auto-run in `gen-mockup`.

### Stage 2.5 — generator hardening (2026-06-03)

The first real `--place-id` run (Utrecht Hairstyle, `ChIJ2eOq…`) surfaced that the **system prompt was *instructing* the generator to fabricate verifiable facts** — and a 5-lens, adversarially-verified audit (21 confirmed findings) confirmed the impact: a fake KvK + BTW in the footer, a dead Treatwell booking CTA, a guessed/404 Instagram link, an invented email, and the Google review count laundered into a "230+ tevreden klanten" stat. None were model misbehaviour — prompt lines 52–54 literally said "verzin plausibele …" and `SiteConfig.legal.kvk` was a *required* string.

Fixed (the mockup is now safe to WhatsApp to a real prospect):
- **Prompt** (`mockup-system.ts`) — replaced every "verzin" clause with hard prohibitions: never fabricate KvK/BTW (omit), never invent email/IG/WhatsApp (omit; WhatsApp only for a real 06/+316 mobile), `booking.provider` defaults to `custom` with no invented third-party URL, `about.stats` only from briefed numbers (never convert review count → klanten; no invented jaren/stylisten), paraphrase testimonials with a neutral anonymous byline, never invent lat/lng. Added a layout anti-default (rating < 4.5 / mixed reviews → `atelier`, not `studio`) and an accent-contrast (≥3:1 vs ink **and** surface) + reserve-metallics rule.
- **Schema** — `legal` is now `.default({})` and `legal.kvk` optional, so the model can omit legal entirely without a wasted retry (was a required string forcing a fake).
- **Pipeline** (`placeToBrief`) — carries the **real** `lat`/`lng`/`rating`/`reviewCount` through as structured `SalonBrief` fields (`briefToMessage` surfaces them as authoritative grounding), and emits "Contact-aanwijzingen" notes (vast nummer → geen WhatsApp; geen website → verzin geen e-mail).
- **Variants** — all three footers suppress the KvK/BTW line when absent and strip a leading `@` before building the Instagram URL (fixes the `instagram.com/@handle` 404); all three booking sections fall back to `tel:` (then `#`) instead of a dead anchor; studio guards the `provider` label / neon the booking copy against `custom`.
- **Verified end-to-end**: regenerated Utrecht Hairstyle → valid config in **1 attempt**, now `layout: atelier`, `legal: {}`, `booking.provider: custom`, **real coordinates** `52.0936634, 5.1192292`, no email/IG/whatsapp, neutral testimonial bylines. Rendered page contains zero fabrications (KvK/BTW/treatwell/instagram.com/"230"/"tevreden klanten" all absent) and the booking CTA resolves to `tel:+31302685951`. Also fixed an SSR bug found en route: the mock app 500'd in dev because Astro's node adapter externalized the raw-TS `@revivo/*` packages → `ssr.noExternal: [/^@revivo\//]`.

### Stage 2.6 — real-data sourcing (Treatwell) + facts passthrough (2026-06-03)

Berend checked the first real Utrecht Hairstyle mockup against the salon's actual Treatwell page and found it wrong on nearly everything the LLM had invented: prices (balayage €165 vs real *from* €70), services (invented Keratine/Olaplex; missed the salon's **threading** specialty), **no team**, reputation hidden (we showed nothing; reality 4,7★/2.907), wrong hours (Google's, not the salon's), generic paraphrased reviews, and picsum photos. Root cause: Google Places is too thin a source, so the model filled the gaps. The fix is to source the salon's **real** facts and stop the model from inventing them.

Built (Phases 1–3 of the real-data plan; **facts deterministic, voice LLM**):
- **`@revivo/sourcing/treatwell.ts`** — a pure fetch+parse scraper (no LLM, no new dep). A Treatwell salon page is server-rendered and embeds a `window.__state__` JSON blob (+ schema.org JSON-LD), so extraction is **deterministic JSON parsing**, not an LLM call. `fetchTreatwellListing` → `RawListing`; `treatwellListingToFacts` → a validated `ListingFacts` (services+prices+durations, hours, team incl. per-stylist rating/sentiment, reputation, curated reviews, photos, phone, geo); `listingFactsToBrief` builds the brief. JSON-LD is a redundant fallback for the scalar fields.
- **`@revivo/shared`** — new `ListingFacts` schema (reuses SiteConfig's own sub-schemas); added optional `team` (`TeamMember`) + `reputation` to `SiteConfig` (additive — every existing config/row still validates; variants render-if-present).
- **`@revivo/llm`** — `generateMockup(brief, client?, facts?)`: when facts are present it (a) surfaces them as an authoritative "ECHTE GEGEVENS" grounding block so the model's **voice** (brand/colour/headline/about/captions/layout) fits the real salon, then (b) `applyListingFacts` deterministically **overwrites** the factual fields (services, hours, team, reputation, testimonials=real reviews, contact.phone, booking=real Treatwell URL, location, **photos**) — the model's invented facts are discarded, so drift can't reach the mockup. The prompt tells the model to omit team/reputation/testimonials (filled from facts). Also added `stripStrayNullsInPlace` (a pre-validation pass that drops stray `null`s the model emits for optional fields — preserving the one legitimately-nullable `price`).
- **Photos folded in** — Treatwell's CDN photo URLs carry no API key, so the Google photo-proxy isn't needed for this path. `applyListingFacts` fills hero (≤4) + gallery from the real photos, **sized dynamically** to however many exist; picsum survives only when there are none.
- **CLI** — `--treatwell <url|slug>` (standalone, or combined with `--place-id` to borrow Google's postcode + coords). New `source: "listing"` (migration `20260603120000_mockups_source_listing.sql` widens the CHECK; `MockupSource` union updated).
- **Verified**: a deterministic scraper test hit **33/33** ground-truth assertions for Utrecht Hairstyle (real prices, 7 categories/26 services incl. threading, correct hours, 5 team members with Carmen's low-count rating suppressed, 2★ review dropped, 15 photos, 4,7★/2.907, geo). A `--dry-run --treatwell` offline pass produced a valid config with every fact correct. A **real** LLM run (`anthropic/claude-sonnet-4.5`, 3.2k in / 2.0k out) produced grounded voice (tagline "20 jaar internationale expertise", about prose weaving Nael's Dubai/Libanon history + threading + "voor iedereen", `about.stats` correctly absent) with all factual fields overwritten by the truth. All of `@revivo/shared|sourcing|llm|db` typecheck clean.
- **Renders now** (existing sections): services/prices, testimonials (real reviews), hours, hero+gallery (real photos), booking (real URL), contact, location. **Pending render**: `team` + `reputation` need a new section in each of the 3 variants (Phase 5).
- **Known gap**: Treatwell has no postcode, so treatwell-only mode leaves `location.postcode` to the model (plausible but a guess); combine with `--place-id` for the real one.

### Stage 2.7 (Phase 6) — verification: screenshot-vision measured unreliable → text fabrication check (2026-06-05)

Built the verification instrument, then measured it — and the measurement retired the planned design.

- **Vision capability added to `@revivo/llm`** — `CompleteOptions.images` (OpenAI-style content parts), `createVisionClient()`, `VISION_LLM_MODEL` (default `qwen/qwen3.7-plus`; confirmed multimodal on OpenRouter, 1M ctx, ~$0.40/$1.60 per Mtok). No provider-SDK leak; same model-agnostic seam as the text client.
- **New `@revivo/verify`** — `playwright-core` full-page screenshots (resolves the *cached* Chromium headless-shell so it never downloads; JPEG q82 @ DPR1 to fit the vision payload cap), a **find→confirm** vision judge (`verifyMockupAgainstListing`), and `serveMockups` (spawns the mockups SSR app with Supabase env stripped → local `examples/generated` JSON → real per-variant `/v/<layout>` CSS). Verification is its OWN `verify-mockup` command, **not** `--verify` on gen-mockup — that would make a `llm→verify→llm` package cycle.
- **The measurement (Utrecht Hairstyle):** on a mockup whose every fact was hand-verified CORRECT against Treatwell (price €55, hours incl. Sunday closed, review count 2909, phone = the real scraped number), the screenshot-vs-screenshot judge reported **3 critical/major "mistakes" — all 3 false positives.** It misread €55 as €35, misread the Sunday "gesloten" cell as "geopend", and misread the phone digits *and* flagged the real number as "invented" (Treatwell renders no phone on the page). The adversarial confirm pass correctly killed ONE FP (a review's "bakkie koffie", present on both pages) but shares the find pass's OCR blind spots on a ~19000px downscaled page. Root causes: small-text OCR misreads, non-determinism, and "not rendered on the listing screenshot" ≠ "invented in reality."
- **Decision (with Berend):** do NOT promote vision to a runtime `--push` gate — per the "measure before complexity" anchor, the deterministic scrape is the accurate part; the comparator is the unreliable part. `@revivo/verify` is kept as a manual spot-check only (its CLAUDE.md carries the FP caveat).
- **Built instead — `checkAboutFidelity` (`@revivo/llm/src/check-about.ts`):** the durable replacement. Facts are pinned deterministically, so the only place the LLM can be confidently wrong is the about-prose; that's a TEXT problem. A short text-LLM call compares the about-copy (+ tagline/headlines) against the salon's real `facts.description` + known structured facts and flags concrete unsupported claims (music/drinks/awards/year/experience/backstory). Auto-run in `gen-mockup` (warn, don't block). **Validated both directions:** the real Utrecht about → `clean` (0 claims); an injected "Spaanse muziek" + "sinds 1995" + "2019 award" about → `fabrication` (3 claims, correctly classified). This is exactly the "Spaanse muziek" class the Phase 6 vision pass was meant to catch — now caught reliably and ~for free.

### Roadmap Phase A — correctness floor (2026-06-09)

Cheap, deterministic hardening of the moat's load-bearing scraper, before any batch volume multiplies a silent bug across every prospect. No new infra; no LLM/vision at runtime.

- **A1 — scraper regression harness + golden fixtures.** Stood up **vitest** (the repo had zero tests). Split a pure `parseTreatwellHtml(html, sourceUrl)` out of `fetchTreatwellListing` so the extractor is unit-testable offline. Captured the real Utrecht Hairstyle page (751 KB) into `packages/sourcing/test/fixtures/treatwell/` and froze a golden `ListingFacts` snapshot + targeted assertions (every price, all 7 hours rows, 5-person team, reputation 4,7★/2913, ≥4★ review dedup, key-free photos). A `no-state.html` fixture proves the JSON-LD fallback recovers scalars, warns, and **omits** services/team/reviews rather than inventing them. Regression-catching verified: corrupting a parse helper fails with a crisp field-targeted diff. `pnpm -r test` runs fully offline (no network, no keys).
- **A2 — structured cross-source fidelity check (`crossCheckListing`, `packages/sourcing/src/fact-check.ts`).** A Treatwell page carries the salon's scalars twice (`window.__state__` + JSON-LD); we re-extract each source independently and flag any disagreement on name/geo/rating/reviewCount/hours/photos — exactly the failure a layout change causes (state parse breaks, JSON-LD intact). ### Batch audit + generator hardening (2026-06-10)

First real batch (13 mockups: 3 + 10) audited by a 118-agent verification fleet — every mockup independently compared against its salon's LIVE Treatwell listing (rendered page, deliberately NOT our own scraper) with 3-voter adversarial re-checks per claimed discrepancy (5 false positives killed). Result: 4 send-ready, 9 held — but the holds collapsed to **three systemic generator bugs**, all fixed the same day:

1. **Required postcode forced invention.** `location.postcode` was required while Treatwell carries no postcode → the model HAD to invent one → 7/13 mockups had a verifiably wrong postcode. Fix: postcode is `.optional()` (schema + prompt "ALLEEN uit de briefing, anders WEGLATEN" + stub), all three variants render-if-present, maps queries `filter(Boolean)`. The omit-don't-invent list now explicitly includes postcode.
2. **Scraper used Treatwell's promo price.** `servicesFromState` read `minSalePriceAmount` — Treatwell's own temporary discount (off-peak slots; one audited salon had every off-peak service at 0.9× the real price, another showed 55%-promo prices as standard). Fix: `minFullPriceAmount` first; golden snapshot deliberately updated (the fixture itself contained one off-peak promo: Kleuren - Eén kleur €33,75 → real €45).
3. **"vanaf" was stripped from from-prices.** `cleanName` dropped the "… vanaf" suffix and only the range-low survived → "Highlights vanaf €7–…" rendered as flat "€7". Fix: additive `ServiceItem.from` flag (set on named-vanaf or genuinely ranged items), rendered as "vanaf €X" by all three variants; NL decimal rendering fixed alongside (€67,50, never €67,5/€67.5).

Plus prompt + about-check guardrails for the two voice-level audit findings: location-character claims ("hartje Utrecht", "loopafstand van CS" for salons 3 km out in Zuilen) and menu-foreign specialisms (an invented "wimperextensions"). The hardening review (37 agents, 8 confirmed / 3 refuted) then closed the escape paths: `transitNotes` — the field where invented transit claims historically always landed — is now deterministically DROPPED by `applyListingFacts` in facts mode AND included in the about-check's text; the from-flag and the name-strip share one regex; a missing `minFullPriceAmount` warns loudly instead of silently reverting to promo prices; an off-enum claim `kind` can no longer void the whole check (`.catch("other")`). Audit side-intel: 3 of 13 salons have no website at all (strongest fits); most others have dated/thin sites — recorded per lead in the audit output.

> **Deploy ordering (operational):** the deployed mock app bundles the schema at build time — **redeploy `apps/mockups` BEFORE regenerating** configs without postcode, or the old bundle's required-postcode parse hard-500s the live mock URLs (and silently strips `from` flags).

All 14 mockups (13 audited + the stub) were regenerated post-deploy with the fixed generator and live-verified: postcode omitted 14/14, transitNotes gone, vanaf-prijzen rendering on every page (1-52 items each), 0 picsum, the three audit holds resolved (Lina full prices, Parbella/Metamorfosa clean about).
