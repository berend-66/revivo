How we collaborate with Git

The whole thing in one picture. Never work directly on main — always branch, then merge back through a Pull Request.

⭐
The one rule to never forget

Before you start anything new: switch to main and pull, so you're up to date with the latest work. Then make a fresh branch. git checkout main → git pull

Two kinds of branches

main protected

The single source of truth. It's always working and always deployable. You never edit it directly — code only enters main through a reviewed Pull Request.

your branch your sandbox

A copy of main where you do your work. Break things, experiment, commit freely — main stays safe. Name it for what you're doing, e.g. add-contact-form.

What that looks like over time

Blue = main (always good) · Green = your branch. Steps ② branch and ④ push don't change the shape — they just create the green line and upload it to GitHub.

The loop — repeat for every new task

1

Get up to date

Start on main and pull the latest. This is the step everyone forgets — don't.

git checkout main   &&   git pull
2

Make your branch

Branch off main so your work is isolated.

git checkout -b add-contact-form
3

Work & save your progress

Edit files, then save snapshots with commits. Commit often — each one is a checkpoint you can return to.

git add .   &&   git commit -m "add contact form"
4

Push to GitHub

Upload your branch so it lives online and I can see it.

git push -u origin add-contact-form
5

Open a Pull Request

On GitHub, click “Compare & pull request”. This asks to merge your branch into main. I review it, we discuss if needed, then it gets merged. ✅

6

Back to step 1

Once merged, your work is part of main. For the next task, pull main again and make a new branch. Never reuse an old branch.

Why we bother with all this

  • main never breaks. The live site builds from main, so it has to always work. Branches keep half-finished work away from it.
  • Pull first = no conflicts. Starting from the latest main means you're not building on top of old code, which avoids painful merge messes later.
  • Pull Requests are our checkpoint. Nothing reaches main without a second pair of eyes — that's where we catch bugs before customers do.
  • Small branches are easy. One task = one branch = one PR. Easy to review, easy to undo if something's off.
Stuck? Just ask before pushing — undoing is always easier than you think. 🙂
